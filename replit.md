# TTB Label Review PoC

AI-powered compliance tool that checks alcohol beverage labels against TTB mandatory labeling requirements using Claude Vision, returning PASS/FAIL/NEEDS REVIEW with field-level detail.

## Run & Operate

- `pnpm --filter @workspace/api-server run dev` — run the API server (port 8080 by default; reads `$PORT`)
- `pnpm --filter @workspace/ttb-label-review run dev` — run the React frontend
- `pnpm run typecheck` — full typecheck across all packages
- `pnpm run build` — typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks and Zod schemas from the OpenAPI spec
- Required env: `SESSION_SECRET` — Express session secret (set in Replit secrets)
- Required env: `AI_INTEGRATIONS_ANTHROPIC_BASE_URL` + `AI_INTEGRATIONS_ANTHROPIC_API_KEY` — auto-set by Replit Anthropic integration
- Required env: `DATABASE_URL` (+ `PGHOST`, `PGPORT`, `PGUSER`, `PGPASSWORD`, `PGDATABASE`) — auto-set by Replit PostgreSQL integration

## Stack

- pnpm workspaces, Node.js 24, TypeScript 5.9
- API: Express 5 + multer (file uploads) + pino (logging)
- AI: Anthropic `claude-sonnet-4-6` via `@workspace/integrations-anthropic-ai`
- Frontend: React 19 + Vite + Tailwind CSS + shadcn/ui + TanStack Query + Wouter
- API contract: OpenAPI 3.1 → Orval codegen → React Query hooks + Zod schemas
- Build: esbuild (ESM bundle)
- Session store: PostgreSQL via Drizzle ORM (`@workspace/db`) — results persist across restarts

## Architecture decisions

- **Claude extracts, engine decides** — Claude Vision returns raw verbatim text; all pass/fail logic lives in `compliance-engine.ts` in TypeScript, not in the AI prompt. This keeps compliance rules auditable and testable.
- **Multi-image in one Claude call** — When front + back labels are both provided, both images go in a single Anthropic message (not two calls). Claude is instructed to extract fields across all provided images. This correctly handles Government Warning on back, brand on front.
- **SFOV check is SPIRITS-only** — 27 CFR 5.64 same-field-of-vision applies only to distilled spirits. `sameFieldOfVision` is `null` for WINE and MALT.
- **Wine country of origin always required** — 27 CFR 4.32(a)(3) requires country of origin on ALL wine labels, even domestic. Domestic spirits/malt do not require it.
- **Sulfite absent = NEEDS_REVIEW, not FAIL** — Lab must confirm < 10 ppm before a "no sulfites" claim is valid. The engine cannot determine this from an image alone.
- **CSV import is client-orchestrated** — Each CSV row: format label text → `POST /generate-preview` (SVG) → canvas SVG→PNG → `POST /upload`. Session ID is shared across all rows so results appear in one session.
- **OpenAPI-first codegen** — Never edit `lib/api-client-react/src/generated/` or `lib/api-zod/src/generated/` by hand. Always update `openapi.yaml` first, then run codegen.

## Product

Four upload modes on the main page:
1. **One Label** — drag-and-drop single image; optional back label toggle for two-panel products
2. **Multiple Labels** — sequential batch queue, all results in one session
3. **Generate Label Image** — paste label text → Claude generates SVG label → compliance check
4. **CSV Import** — upload `applications.csv` → AI generates label images per row → batch compliance check → unified session report

Results dashboard shows Pass/Fail/Review summary with filterable table and CSV export. Label detail page shows per-field breakdown, Government Warning side-by-side comparison, wine-specific fields (appellation, sulfite declaration), and expandable "How to Fix" remediation cards.

Supported beverage types and their specific checks:
- **SPIRITS (27 CFR Part 5)** — ABV always mandatory; same-field-of-vision check (brand + class + ABV on same panel)
- **WINE (27 CFR Part 4)** — ABV mandatory ≥7%; country of origin always required; appellation required when varietal/vintage present; sulfite declaration checked
- **MALT (27 CFR Part 7)** — ABV not mandatory (unless strength claim made); no SFOV requirement

## Gotchas

- **Run codegen after any OpenAPI change** — `pnpm --filter @workspace/api-spec run codegen`. Missing fields in the frontend types almost always mean a stale codegen run.
- **Do not call `pnpm dev` at workspace root** — no root `dev` script exists. Use the Replit workflows or `pnpm --filter @workspace/<name> run dev`.
- **`upload.fields()` not `upload.single()`** — the `/upload` route uses `multer.fields([{name:"file"},{name:"backFile"}])` to support optional back label. `req.files` is a Record, not a single file.
- **CONFIDENCE_THRESHOLD = 0.6 globally; SFOV uses 0.75** — defined as constants at the top of `compliance-engine.ts`.
- **Levenshtein threshold = 3** for brand name fuzzy match (not 2 as the old README stated).
- **`label-analyzer.ts` signature** — `analyzeLabel(images: LabelImage[], fileName: string, options)`. The `images` array can have 1 or 2 entries.
- **Session store is PostgreSQL-backed** — results survive server restarts. Schema: `label_results` table with `label_id` PK, `session_id` (indexed), `result` JSONB, `analyzed_at`.
